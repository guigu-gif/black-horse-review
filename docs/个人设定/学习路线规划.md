## 二、为什么这样学（底层逻辑）

你的问题是：框架理解强，代码实践弱，基础API不熟。

所以策略是：**用项目驱动基础学习**，而不是先把基础学完再做项目。
原因：
- 先学基础再做项目 → 基础学完就忘，不知道用在哪，容易放弃
- 先做项目边补基础 → 遇到不懂的马上有动机去查，记得牢，学得快

学习顺序遵循你的习惯：**底层原理 → 整体框架 → 动手实践 → 费曼阐述 → 间歇复习**

### 阶段一：假期最后10天（2026-02-21 ~ 2026-03-01）
**每天投入：4-5小时**

**目标**：建立Java代码手感，能看懂并抄写项目代码

**为什么从这里开始**：
你说Java基础和API是最薄弱的，但不能死背API，要通过项目代码来理解。
黑马点评的代码已经在那里了，最高效的方式是直接读它、抄它、改它。

**每天安排**：
- 上午：理解一个Java基础概念的底层原理（为什么存在，解决什么问题）
- 下午：在黑马点评项目里找到对应的代码，读懂每一行，写注释，照着抄一遍

**Java基础学习顺序（按项目中出现频率排）**：
1. 集合框架：List、Map、Set → 为什么要有这三种，底层数据结构是什么

3. 异常处理：try-catch-finally，为什么要有异常机制
4. 常用API：String操作、时间类、Random
5. Lambda表达式：为什么出现，简化了什么

**阶段结束标准**：
能看着黑马点评代码，说出每一行在干什么，并照着抄出来不报错

## 八、项目理解：黑马点评是什么

### 项目定位
黑马点评本质是一个普通的Spring Boot + MyBatis + MySQL的传统点评平台。
Redis不是主角，是解决高并发和性能问题的工具。
学这个项目的目的：**学会在真实业务场景里，在哪里用Redis、为什么用、怎么用。**

### 后端工程师的职责
- 接收前端传来的参数
- 写业务逻辑（查数据库、处理数据、调用工具）
- 返回结果给前端
- 前端怎么渲染不是后端的事

### 项目整体工作流程（一个请求的完整生命周期）

```
用户操作（手机/浏览器）
        ↓
    前端发HTTP请求
        ↓
  Controller层（接收请求，校验参数）
        ↓
  Service层（核心业务逻辑）
        ↓         ↓
   Redis缓存    MySQL数据库
  （查缓存）    （查不到再查这里）
        ↓
  返回结果给前端
```

### Redis在哪里介入、为什么

| 场景 | 没有Redis的问题 | Redis怎么解决 |
|------|----------------|--------------|
| 查店铺详情 | 每次都查MySQL，高并发下数据库扛不住 | 第一次查MySQL后存Redis，后续直接读Redis |
| 秒杀抢券 | 多个请求同时扣库存，出现超卖 | 用Redis的原子操作控制并发 |
| 用户登录状态 | Session在分布式环境下不共享 | 用Redis存Token，所有服务器共享 |
| 点赞排行榜 | MySQL做排序性能差 | 用Redis的ZSet天然支持排序 |

## 九、工具扫盲（大家都知道但你还不知道的）

### Git — 版本控制
解决的问题：代码改坏了能回退，多人协作不冲突。
你需要掌握的7个命令（覆盖90%日常使用）：
```
git init        # 初始化仓库
git add .       # 把改动加入暂存区
git commit -m   # 提交一个版本
git push        # 推送到远程（GitHub）
git pull        # 拉取最新代码
git branch      # 创建/查看分支
git merge       # 合并分支
```
学习成本：半天能搞定基础。

### GitHub — 代码托管平台
就是把你的Git仓库放到网上。
对你的意义：简历上附GitHub链接，面试官能看到你的代码，增加可信度。

### 部署 — 让别人能访问你的项目
流程：买云服务器 → 装环境（Java/MySQL/Redis）→ 上传项目 → 启动
Docker的作用：把环境打包成一个箱子，搬到哪里都能跑，省去装环境的麻烦。
学习成本：项目做完后一周能搞定基础部署。

### 什么时候学这些
项目做完（阶段二结束）后统一学，一周搞定Git+GitHub+Docker基础部署。
现在不用管，先专注项目本身。
